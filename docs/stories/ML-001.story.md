# ML-001 — Improve app theming (complete Dark Mode)

## Details

### Problem

The application ships with both Light and Dark modes, but the Dark mode variant is incomplete. Several UI components and areas render incorrectly or with inconsistent styling when Dark mode is enabled (see attached screenshot). Issues observed include:

- Incomplete or missing dark backgrounds for form fields, cards and modal panels.
- Poor contrast on labels, input placeholders and helper text (making them hard to read).
- Disabled or pending states (badges, progress indicators) use colours optimized for Light mode only and are nearly invisible on dark backgrounds.
- Button variants (primary/secondary/ghost) and their hover/focus states are inconsistent between modes.
- Some components assume a light background and show faint borders or shadows that disappear in dark theme.

Where it manifests:

- Workflow runner pages (e.g., `src/app/w/[slug]/page.tsx`) — form panels and preflight cards.
- Global UI components under `src/components/` including `FormFields.tsx`, `UploadCard`, `MediaDisplay.tsx`, `CacheDashboard.tsx`, and `JobStatus.tsx`.
- Any pages using the standard container/background wrappers (global `globals.css` and layout at `src/app/layout.tsx`).

### Impact

- Users who prefer Dark mode get a broken or low-quality experience leading to confusion and reduced trust.
- Accessibility issues: insufficient contrast may fail WCAG guidelines and lead to support requests.
- Visual regressions make the UI feel unfinished — harms product perception.
- Potential bugs where invisible UI elements (controls, error messages) prevent successful workflow submissions.

### Scope

In scope:

- Audit and fix UI styles to ensure a complete, consistent Dark mode experience across the web app.
- Update global CSS variables (Tailwind configuration / CSS custom properties) to supply proper dark-mode tokens.
- Fix component-level styles in `src/components/` that hardcode light colours or rely on implicit backgrounds.
- Add Z-index / shadow and border adjustments that work on dark backgrounds.
- Add unit and component tests where appropriate to assert theme classes and contrast of key components.
- Update documentation in `docs/` and add a short developer note describing theming tokens and how to add new components.

Out of scope (for this story):

- Rewriting the entire design system or changing existing colour palette beyond dark-mode parity.
- Mobile native or Electron packaging changes (this repo targets web). Those can be follow-ups.

---

## Acceptance criteria

- Given the user toggles the app to Dark mode, when any page loads, then all form fields, cards, buttons, badges, and helper text render with the correct dark-mode tokens and maintain minimum contrast ratios.
- Given a workflow runner page (for example `w/[slug]`), when Dark mode is enabled, then the Model Preloading and Model Requirements cards match the light-mode layout but with dark backgrounds, visible borders/shadows, and accessible text colours.
- Given buttons in primary/secondary/ghost variants, when hovered/focused/disabled in Dark mode, then visual states are consistent, accessible and match design tokens.
- Given an input or textarea with placeholder/helper text, when rendered in Dark mode, then placeholder and helper text meet WCAG AA contrast against the input background.
- Given failing preflight or error messages, when displayed in Dark mode, then error states show clearly with accessible colours and i18n keys present.
- All modified components include unit or component tests asserting they render expected class names or inline styles for both Light and Dark modes.
- Documentation: `docs/stories/ML-001.story.md` exists (this file) and `docs/` contains a short `docs/stories/README.md` update linking to the theming guidelines (or a note added to existing docs if present).

## Notes & Implementation Hints

- Prefer using Tailwind's class-based dark mode (if project uses it) and rely on CSS variables for colour tokens to make it easy to update a single token for both modes.
- Example token set (CSS custom properties) to add in `globals.css`:

```css
:root {
  --bg: #ffffff;
  --panel: #f8fafc;
  --text: #0f172a;
  --muted: #6b7280;
  --primary: #6366f1;
}

:root.dark,
.dark {
  --bg: #0b0b0f;
  --panel: #0f1724;
  --text: #e6eef8;
  --muted: #9aa6b2;
  --primary: #7c83ff;
}
```

- Use `getComputedStyle()` in tests to assert colours when running headless JSDOM-based checks.
- When updating components, avoid inline hard-coded hex colours; use token variables or Tailwind `text-...`/`bg-...` tokens consistent with the project's conventions.
- For accessibility, use `axe` or `jest-axe` in component tests to assert contrast for critical UI pieces.
