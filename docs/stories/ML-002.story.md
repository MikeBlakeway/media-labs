## Details

### Problem

Workflow pages expose a large number of input fields. Most users only need to edit a small subset of these for normal usage. The current UI shows all inputs by default, which increases visual complexity, causes cognitive load, and makes the form harder to scan. This is particularly visible on workflow runner pages where advanced model parameters, scheduler options, or rarely changed toggles are displayed alongside primary inputs.

### Impact

- Increased cognitive load and slower form completion for users running workflows.
- Important, frequently-used fields (prompts, uploads, output size/length) can be harder to find among advanced options.
- Higher risk that users accidentally change rare parameters, causing confusing results and support issues.

### Scope

In scope:

- Update workflow runner pages under `src/app/w/[slug]/` so that only the essential inputs are visible by default. Essential inputs are:
  - Text prompt blocks (Positive Prompt, Negative Prompt)
  - Image / Video upload inputs
  - Output size/length inputs (Width, Height, Frame Count, Video Length)
- Move all other inputs into a collapsible "Advanced settings" area that is closed by default and can be toggled open with an onClick handler.
- Preserve existing form state and validation; collapsing advanced settings must not clear or lose values.
- Add a small persistent summary line in the collapsed header (e.g., "3 advanced settings set") so users can see if non-default options are present.
- Ensure the solution is accessible (keyboard operable, focus management when expanded, screen-reader labels).

Out of scope:

- Changing which specific parameters are considered essential beyond the set listed above (this can be a follow-up configurable task).
- Large redesigns of the runner page layout or moving inputs to separate pages — keep changes minimal and incremental.

---

## Acceptance criteria

- Given a workflow page, when it loads, then the form shows only the essential inputs (prompts, uploads, width/height/frame count/video length) and the rest are hidden inside a collapsed "Advanced settings" section.
- Given an advanced input has a non-default value, when the section is collapsed, then the collapsed header shows a concise summary (like "3 advanced settings set").
- Given a user toggles the Advanced settings header via keyboard (Enter/Space) or mouse click, when the section opens, then focus moves to the first field inside and screen readers announce the expanded state.
- Given the Advanced settings are collapsed and a user submits the form, when the form validates, then values inside collapsed advanced inputs are included in the submission payload (no data loss).
- Given the user expands and then collapses the Advanced settings, when they reopen it later in the same session, then previously-entered values are preserved.
- All modified components include unit and component tests asserting collapsed state, summary counts, keyboard interaction, and that values are preserved across toggles.

## Notes & Implementation Hints

- Prefer creating a small reusable `CollapsibleSection` component rather than adding ad-hoc show/hide logic in multiple forms.
- Use CSS transitions for the height/opacity but avoid layout shifts that harm keyboard users; prefer toggling max-height with reduced motion fallback.
- For non-default counting, compare each field against its schema default (if present) or the default value from the form initial state.
- If the app uses React Hook Form, collapsing should not unmount fields (or use `shouldUnregister: false`) to preserve state; otherwise persist values in a parent state object.
- For accessibility, ensure the toggle button has `aria-expanded`, `aria-controls` pointing to the container id, and that the container has `role="region"` with `aria-labelledby`.
