## Details

### Problem

At present generated outputs (images, videos, metadata) exist only for the duration of the current browser session. When the user closes the app, all references and UI history are lost. Although the binary output files are uploaded to Backblaze B2 and served via signed URLs, the application does not maintain a server-side index or a database of generated items to rehydrate user sessions or show previously generated results.

### Impact

- Users lose access to previously generated content when they close their session, leading to poor UX and potential data loss.
- Users cannot search, filter or manage their past results through the app (no history, favourites, or export flows).
- Relying only on signed URLs in the client is fragile: signed URLs expire and there is no canonical mapping between UI metadata and bucket objects.

### Scope

In scope:

- Design and add a server-side persistent storage layer (database) to store metadata about generated outputs, linking to object locations in Backblaze B2 (signed URL or canonical B2 key).
- Migrations and data model: simple outputs table with fields like id, user_id (nullable for anonymous), workflow_slug, created_at, outputs[] (array of objects: key, mime, size, preview_url), params (workflow parameters), status, and optional tags/notes.
- API endpoints to GET/POST outputs: allow saving new outputs when a workflow completes, listing user's previous outputs, and retrieving metadata for display.
- Add a background job or startup routine to optionally scan B2 bucket prefixes used by the app and reconcile objects into the DB (safe, idempotent, optional feature behind a flag).

Out of scope:

- Full multi-tenant account management and long-term retention policies (this can be a follow-up).
- Migration of old anonymous results unless a migration job is explicitly requested.

---

## Acceptance criteria

- Given a workflow finishes and produces output files, when the workflow result is finalised, then the UI/server saves a record in the database that includes metadata and a canonical reference to the stored object(s).
- Given a returning user (same session or authenticated), when they open the app, then previously saved outputs are listed and accessible for preview/download.
- Given a signed URL expires, when the user attempts to view an older result, then the server can re-generate a fresh signed URL for the B2 object and return it via an API endpoint.
- Given the optional bucket scan is enabled, when the service runs the scan, then it creates DB records for objects not already present without duplicating or overwriting existing entries.
- All new APIs are documented and covered by unit/integration tests verifying correct behaviour and edge cases (missing object, expired URL, permission errors).

## Notes & Implementation Hints

- For a minimal implementation, start with a simple Postgres (or SQLite for local dev) table `outputs` and use an ORM like Prisma to simplify schema/migrations.
- When saving outputs, store the canonical B2 object key (not the signed URL). Signed URLs should be generated on demand by the server API when the client requests preview/download.
- Respect privacy: if outputs are user-specific, link `user_id` where available. For anonymous users, either keep records without user linkage or provide an opt-in account-link flow later.
- The B2 scan should be optional and idempotent. Use object ETag/LastModified to detect duplicates and avoid creating multiple DB entries for the same object.
- Rate-limit bucket scans and provide an admin-only API to trigger manual scans.
