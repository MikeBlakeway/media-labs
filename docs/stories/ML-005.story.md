# ML-005 — Add stable layout with navbar and footer navigation

## Details

### Problem

The application currently lacks consistent navigation elements. Users must rely on browser back buttons or manually edit URLs to navigate between different sections of the app. There is no persistent header/navbar or footer to provide clear navigation paths, making the app feel incomplete and potentially confusing for users who want to explore different workflows or access various app sections.

Where it manifests:

- All pages lack consistent navigation elements
- Users get "trapped" on workflow pages with no clear way to return to home or browse other workflows
- No branding or app identity visible across pages
- Missing links to important sections like admin, manage, debug, etc.

### Impact

- Poor user experience: navigation is unintuitive and relies on browser controls
- Users may not discover all available features due to lack of navigation
- App feels unprofessional without consistent branding and layout structure
- Potential accessibility issues for users who rely on keyboard navigation or screen readers expecting standard navigation landmarks

### Scope

In scope:

- Design and implement a persistent navbar component with:
  - App branding/logo
  - Main navigation links (Home, Workflows, Manage, etc.)
  - Responsive design for mobile and desktop
  - Accessibility features (ARIA landmarks, keyboard navigation)
- Add a footer component with:
  - Secondary navigation links
  - App information (version, links to docs, etc.)
  - Consistent styling with the navbar
- Update the root layout to include navbar and footer on all pages
- Ensure navigation works with Next.js App Router and doesn't interfere with existing page functionality
- Add proper active state indicators for current page/section

Out of scope:

- Complete design system overhaul (use existing design tokens and patterns)
- User authentication/profile UI in navbar (can be a follow-up)
- Advanced mobile menu features (hamburger menu can be simple initially)

---

## Acceptance criteria

- Given any page in the application, when it loads, then a consistent navbar appears at the top with app branding and main navigation links.
- Given a user clicks a navbar link, when navigation occurs, then the active page is visually indicated in the navbar and the correct page loads.
- Given a user navigates via keyboard (Tab), when focusing navbar elements, then all links and interactive elements are accessible and properly announced by screen readers.
- Given the viewport is mobile-sized, when the navbar renders, then it adapts responsively (collapsible menu or stacked layout) without breaking functionality.
- Given any page in the application, when it loads, then a footer appears at the bottom with secondary links and app information.
- Given the user is on a workflow page (`/w/[slug]`), when they click the Home link in navbar, then they navigate back to the main page without losing any unsaved form state warnings if applicable.
- All navigation components include proper ARIA landmarks (`role="navigation"`, `aria-label`) and focus management.

## Notes & Implementation Hints

- Use Next.js `Link` component for client-side navigation to maintain SPA behavior.
- Consider using `usePathname()` from `next/navigation` to determine active nav states.
- For responsive navbar, start with a simple approach (hide/show links) before implementing complex hamburger menus.
- Ensure footer is properly positioned (consider sticky footer if content is short).
- Use semantic HTML: `<nav>` for navbar, `<footer>` for footer, proper heading hierarchy.
- Example navbar structure:

```tsx
<nav role='navigation' aria-label='Main navigation'>
  <div className='navbar-brand'>
    <Link href='/'>Media Labs</Link>
  </div>
  <div className='navbar-nav'>
    <NavLink href='/' active={pathname === '/'}>
      Home
    </NavLink>
    <NavLink href='/manage' active={pathname.startsWith('/manage')}>
      Manage
    </NavLink>
    <NavLink href='/admin' active={pathname.startsWith('/admin')}>
      Admin
    </NavLink>
  </div>
</nav>
```

- Ensure the layout doesn't interfere with existing workflow pages or modal overlays.
